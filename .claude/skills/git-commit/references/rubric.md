# Git Commit — 质量评分标准

对生成的 commit message 从以下 5 个维度评分，每项 0-4 分，总分 20 分。

## 评分维度

### 1. 格式规范性（Format Compliance）— 4 分

是否严格遵循 `<type>(<scope>): <subject>` 格式。

| 分数 | 标准 |
|------|------|
| 4 分 | 格式完全正确：type 小写、scope 合理、半角冒号+空格、subject 无结尾标点 |
| 3 分 | 格式基本正确，有 1 处轻微偏差（如 scope 括号多了空格） |
| 2 分 | 存在 2 处格式问题（如用了全角冒号、忘了空格、type 大写） |
| 1 分 | 多项格式错误，偏离规范格式 |
| 0 分 | 未使用规范格式，为自由文本 |

### 2. type 准确性（Type Accuracy）— 4 分

type 是否准确反映改动性质。

| 分数 | 标准 |
|------|------|
| 4 分 | type 完全匹配改动性质（新增→feat、修复→fix、配置→chore 等） |
| 3 分 | type 基本正确，但存在可商榷的选择（如重构用了 feat） |
| 2 分 | type 明显不匹配（如新增功能标为 chore） |
| 1 分 | type 错误但有部分关联性 |
| 0 分 | type 完全错误或未填写 |

### 3. scope 合理性（Scope Appropriateness）— 4 分

scope 是否使用真实模块名，是否与改动文件路径/目录一致。

| 分数 | 标准 |
|------|------|
| 4 分 | scope 精准反映改动模块，与目录/包名一致 |
| 3 分 | scope 方向正确但粒度可优化（过大或过细） |
| 2 分 | scope 与改动模块关联性弱 |
| 1 分 | scope 使用了一个不存在的模块名 |
| 0 分 | scope 缺失且本应有明确模块归属，或 scope 严重误导 |

### 4. subject 质量（Subject Quality）— 4 分

subject 是否简洁、具体、可理解，符合中文表达规范。

| 分数 | 标准 |
|------|------|
| 4 分 | 简洁具体（8~30 字），准确描述改动结果，术语使用恰当 |
| 3 分 | 描述基本清晰，但可进一步精简或具体化 |
| 2 分 | 存在空泛描述（如"优化代码"）、长度不合适（过短/过长） |
| 1 分 | 模糊不清，难以从 subject 理解改了什么 |
| 0 分 | subject 缺失，或与改动内容无关 |

### 5. 意图单一性（Single Intent）— 4 分

一次 commit 是否只表达一个主要意图。

| 分数 | 标准 |
|------|------|
| 4 分 | 改动聚焦一个明确意图，commit message 准确反映 |
| 3 分 | 意图基本单一，有少量附带改动但可接受 |
| 2 分 | 改动涉及 2 个不相关意图，应在 subject 中拆分 |
| 1 分 | 改动涉及 ≥3 个不相关意图，commit message 无法准确概括 |
| 0 分 | 大杂烩提交，无法判断主要意图 |

## 通过标准

- 总分 ≥ 16 分（满分的 80%）且
- 第 1 项（格式规范性）≥ 3 分（硬性要求）
- 第 4 项（subject 质量）≥ 2 分

## 评分流程

1. 将改动信息（git diff / 描述）作为输入
2. 将生成的 commit message 作为待评分输出
3. 逐维度评分，同时给出具体证据
4. 汇总总分，判定通过/不通过

## 纠偏检查项

- type 拼写检查：出现 `ath` → 应纠正为 `auth`
- type 拼写检查：出现 `doce` → 应纠正为 `docs`
- 冒号检查：出现全角 `：` → 应纠正为半角 `:`
- 空格检查：冒号后必须有且仅有一个空格
