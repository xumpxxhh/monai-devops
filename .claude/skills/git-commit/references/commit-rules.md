# Git Commit 规则详细说明

基于仓库历史提交风格总结的 commit message 规范。

## 格式约束

- `type` 必填，小写英文
- `scope` 建议填写，使用真实模块名
- `subject` 必填，使用简洁中文描述本次改动结果
- 冒号后保留一个空格
- 一次提交只表达一个主要意图
- 新提交统一使用半角标点：`type(scope): subject`（不要用全角 `：`）

## type 取值

| type | 说明 |
|------|------|
| `feat` | 主类型，功能新增、接口改造、能力增强 |
| `chore` | 工程配置、脚本、依赖、仓库维护 |
| `docs` | 文档更新 |
| `fix` | 缺陷修复（历史较少，但可按语义使用） |
| `refactor` | 重构（不改变外部行为） |
| `temp` | 临时提交（默认不推荐，合并前应整理） |

## scope 命名

优先使用项目内真实模块名，保持与目录/包一致。常见 scope：

- `auth` — 认证授权模块
- `config` — 配置相关
- `ip` — IP 访问控制
- `static` — 静态资源
- `refresh` — 刷新/令牌相关
- `scripts` — 脚本

说明：
- 历史里出现过 `ath`，视为 `auth` 的笔误；后续统一使用 `auth`
- 未出现明显模块归属时可省略 `scope`（如 `feat: ...`、`chore: ...`）

## subject 编写规范

- 说明"做了什么"，不写"在做什么"
- 避免空泛描述，如"调整一下""更新代码"
- 不包含句号、感叹号等结尾标点
- 控制长度，建议 8~30 个字
- 优先使用中文；若是术语可保留英文（如 `token`、`CORS`）

## subject 质量示例

### 好的 subject

- `增加 PKCE 保护并完善授权码流程`
- `调整 IP 访问限制规则`
- `增加跨域缓存头配置`
- `修复密码重置邮件偶发收不到的问题`
- `抽取 token 解析逻辑`

### 不好的 subject

| 示例 | 问题 |
|------|------|
| `调整一下` | 空泛，未说明具体调整了什么 |
| `更新代码` | 过于宽泛，无具体信息 |
| `修复了一个bug。` | 结尾有多余标点，且未说明修了什么 |
| `在优化登录流程` | "在"字多余，应直接说"优化登录流程" |
| `改了改样式` | 口语化，不专业 |

## 历史兼容与纠偏

- 历史中出现过 `doce`，后续统一改用 `docs`
- 历史中出现过全角冒号 `：`，后续统一改用半角冒号 `:`
- 提交前自检一次 `type/scope` 拼写，避免 `ath` 这类误写
